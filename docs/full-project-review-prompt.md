# 全项目实现质量 Review Prompt

你是当前仓库中的高级软件工程师。请对这个项目做一次完整、严格、逐文件的实现质量 review，并在确认问题后完成必要修复。

## 目标

完整 review 当前项目的每一个文件，确认现有实现不是临时实现、不是 hack、不是为了绕过问题的假修复，而是符合当前项目架构、平台约束和长期维护要求的稳定实现。

你需要重点检查：

- 每个文件的职责是否清晰，是否符合现有架构边界。
- 当前实现方式是否是项目语境下的最佳实践，而不是临时方案。
- 代码格式、命名、排版、分层、状态管理、生命周期处理是否存在问题。
- 是否存在隐藏 bug、边界条件遗漏、状态不同步、资源泄漏、线程/Actor 问题、UI 命中区域问题、持久化一致性问题。
- 是否存在过度设计、过度抽象、无意义封装、未来主义设计、复杂度不成比例的问题。
- 是否存在不合理逻辑、重复逻辑、脆弱耦合、魔法数字、隐式副作用。
- 是否存在残存的无用代码、无用测试、无用资源、过期注释、废弃路径、死分支。
- 是否存在测试覆盖不足，或者测试只验证源码字符串而没有覆盖真实行为的风险。
- 是否所有代码都有足够详尽、准确、可维护的中文注释。

## 工作方式

1. 先读取仓库说明文件、项目结构、测试结构和构建配置，理解当前项目的架构和约束。
2. 使用文件清单逐个 review 所有项目文件，不要只抽样检查。
3. 对每个文件记录结论：正常、需要修改、需要删除、需要补充测试、需要补充注释。
4. 涉及 SwiftUI、AppKit、Core Data、文件系统持久化、macOS 窗口行为、Liquid Glass 等平台能力时，先查最新 Apple 官方文档，不要依赖过期记忆。
5. 不要用通用最佳实践覆盖项目已有约定。必须先理解当前代码为什么这样写。
6. 不要做无关重构。每一处修改都必须能对应到明确问题。
7. 不允许用 UI 掩盖数据问题，不允许 silent fallback，不允许 fake success。
8. 如果发现实现明显是临时方案，必须指出根因，并改成稳定、可维护的实现。
9. 如果某个问题风险较高或需要产品判断，先写清楚风险和建议方案，不要擅自扩大范围。

## 注释要求

请为所有需要解释的代码补充详尽中文注释。注释必须解释代码的意图、约束、生命周期、数据流、不变量和容易踩坑的原因。

注释规则：

- 注释写中文。
- 注释要解释“为什么这样做”，不要重复“这行代码做了什么”。
- 对 SwiftUI 状态、AppKit bridge、窗口行为、透明度、玻璃效果、文件持久化、测试替身、边界条件等必须补充清楚。
- 对复杂条件判断、跨层调用、异步流程、文件系统操作、用户可见状态变化必须补充清楚。
- 不要写空泛注释，例如“设置变量”“返回结果”“执行操作”。
- 如果某段代码本身过于复杂，优先简化代码，再补充必要注释。
- 如果发现旧注释已经不准确，必须更新或删除。

## Review 清单

逐文件检查以下问题：

- 文件是否仍然被使用。
- 文件职责是否单一。
- 类型、函数、属性命名是否表达准确。
- public/internal/private 访问级别是否合理。
- 状态来源是否唯一，是否存在重复状态。
- 错误处理是否真实暴露失败，而不是吞错。
- UI 状态是否和数据状态一致。
- 是否存在 hard-coded magic number，是否应该抽成项目已有常量。
- 是否存在重复实现，是否应该复用已有组件或工具。
- 是否存在过早抽象或抽象层级过多。
- 是否存在为了测试而污染生产代码的问题。
- 测试是否覆盖真实行为和关键回归。
- 源码字符串测试是否足够稳定，是否需要补充行为测试。
- 是否存在不必要的依赖、资源或配置。
- 是否存在格式问题、尾随空格、文件编码异常、未使用 import。

## 输出要求

完成 review 后，请输出：

1. 逐文件 review 结果摘要。
2. 发现的问题列表，按严重程度排序，并附文件路径和原因。
3. 已完成的修改列表。
4. 删除的无用代码或资源列表。
5. 补充的中文注释范围和原则。
6. 仍需产品确认或人工视觉确认的问题。
7. 执行过的验证命令和真实结果。

如果没有运行验证，必须明确写：

```text
not run
```

## 修改约束

- 不要一次性重写大文件。
- 不要做无关格式化。
- 不要删除仍可能被使用的代码。
- 不要为了追求“最佳实践”改变已有产品行为。
- 不要引入新依赖，除非已经证明现有工具无法稳定解决问题。
- 不要提交 secrets。
- 每个修改都必须可解释、可验证、可回退。

## 验证要求

修改完成后，至少执行：

```bash
git diff --check
xcodebuild -project Momento.xcodeproj -scheme Momento -destination 'platform=macOS' test
```

如果只改文档，可以只运行 `git diff --check`，但必须说明没有运行完整测试的原因。

最终提交前确认：

```bash
git status --short
git diff --stat
```

提交信息使用：

```text
<type>: <summary>
```

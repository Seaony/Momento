# AGENTS.md

## Mission

你是当前仓库中的高级软件工程师。目标是在遵循现有架构、代码风格与系统设计的前提下，做出正确、最小、稳定、可维护的修改。

优先级：

1. 正确性
2. 稳定性
3. 可维护性
4. 最小改动
5. 可读性

禁止炫技、过度设计、无意义抽象。只修改当前任务必须涉及的代码。

## Testing Policy

不要默认走“先写失败测试，再实现”的 TDD 流程。除非用户明确要求，或者改动涉及高风险数据/状态/文件系统逻辑，否则可以直接实现。

保留并优先维护这些测试：

- 资源库创建、打开、重命名、删除、最近资源库、缺失资源库检测等文件系统和数据生命周期测试。
- 导入、去重、缓存清理、元数据持久化等会影响用户数据安全的行为测试。
- 本地化目录完整性测试。
- 少量粗粒度架构护栏测试，例如原生 Liquid Glass、窗口透明工具栏、主窗口最小尺寸这类容易反复回归的规则。

不要为纯视觉微调新增测试，例如 padding、字号、图标大小、hover 亮度、菜单间距、具体 SwiftUI 视图结构。已有这类过细 source test 应该删除或合并为更粗粒度的架构测试。

修改 UI 时，优先靠最小可信验证：

- 能编译。
- 关键现有测试通过。
- `git diff --check` 通过。
- 用户自行运行 App 查看视觉效果；不要主动启动 App。

## Documentation

涉及 Apple / SwiftUI / AppKit / 第三方 API 的实现细节时，先查官方文档或 Context7。纯文档更新、测试清理、简单字符串调整不需要为了形式而查文档。

## Validation

任何有效修改都必须验证。优先：

1. 最相关的测试
2. 类型检查 / build
3. lint / `git diff --check`

未运行验证时必须写：

```text
not run
```

禁止编造验证结果。

## Git

完成任务并通过基础验证后提交。当前项目偏好是：每次完成一轮用户要求的更新后 commit。

提交格式：

```text
<type>: <summary>
```

禁止 force push、自动 push、提交 secrets、WIP commit。
